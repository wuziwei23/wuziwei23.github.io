实验八报告

实验八 软件部署 —— 实验报告

一、实验目的

综合运用前面各实验所学工具，完整体验 Hexo 静态站点从开发到部署的全过程：一方面通过 GitHub Actions 实现自动化部署（CI/CD）到 GitHub Pages，另一方面手动将站点部署到 Nginx 云服务器（虚拟机），理解两种部署方式的原理与区别。

二、实验环境

  • 操作系统：Windows 11 家庭中文版（本地）+ Ubuntu Server 22.04（虚拟机）
  • 构建工具：Hexo 7.x，Node.js 20
  • CI/CD：GitHub Actions（peaceiris/actions-gh-pages@v3）
  • Web 服务器：Nginx
  • 版本控制 / 托管：Git、GitHub（用户名 wuziwei23）
  • 虚拟机：IP 192.168.80.128，登录用户 wuziwei

三、实验过程

第一部分：GitHub Pages 自动化部署

3.1 准备仓库

创建两个 GitHub 仓库：源码仓库 my-blog（Private，存放 Hexo 源码）与部署仓库 wuziwei23.github.io（Public，托管静态页面），并将本地博客项目关联到 my-blog 后推送到 main 分支。

3.2 配置 GitHub Actions 工作流

在项目下新建 .github/workflows/deploy.yml，定义在推送到 main 时自动执行的流水线：检出源码 → 安装 Node.js 20 → 缓存并安装依赖 → hexo generate 生成静态文件 → 创建 .nojekyll → 通过 peaceiris/actions-gh-pages 部署到 wuziwei23.github.io。


图1：deploy.yml 工作流配置，external_repository 指向 wuziwei23/wuziwei23.github.io

3.3 配置 Token 与 Pages

在 GitHub 生成勾选 repo 权限的 Personal Access Token，添加到源码仓库的 Secrets（名称 PERSONAL_TOKEN）供工作流使用；在部署仓库的 Settings → Pages 中启用 main 分支。此后每次 git push 都会自动触发构建与部署。

3.4 访问上线站点

工作流运行成功后，浏览器访问 https://wuziwei23.github.io ，博客正常显示 Hexo（landscape 主题）首页及「原神」「Hello World」文章。


图2：公网访问 wuziwei23.github.io，自动部署的博客正常显示

第二部分：部署到云服务器（虚拟机）

3.5 安装配置 Nginx

在虚拟机中 apt install nginx，创建站点目录 /var/www/blog，并配置 Nginx 站点将根目录指向该目录后 reload。

3.6 使用 scp 同步静态文件

本地执行 hexo generate 生成静态文件后，通过 scp 将 public/ 下的内容上传到服务器站点目录：

首次连接确认主机指纹并输入密码后，index.html、banner.jpg、style.css、脚本等文件依次以 100% 传输完成。


图3：scp -r 将 public 下的静态文件同步到 192.168.80.128:/var/www/blog/

3.7 访问服务器站点

浏览器访问虚拟机 IP http://192.168.80.128 ，Nginx 正常返回博客页面，与 GitHub Pages 上的内容一致。


图4：访问 192.168.80.128，Nginx 托管的博客正常显示

四、实验结果

测试项 命令 / 操作 结果
自动化工作流 推送 main 触发 GitHub Actions ✅ 成功
GitHub Pages 访问 访问 https://wuziwei23.github.io ✅ 成功
Nginx 站点部署 安装配置 Nginx，站点目录 /var/www/blog ✅ 成功

同一份 Hexo 博客成功通过「自动化 CI/CD → GitHub Pages」和「手动 scp → Nginx 服务器」两种方式完成部署，均可正常访问。

五、知识总结

1. 两种部署方式对比

维度 GitHub Pages（自动） Nginx 服务器（手动）
触发方式 推送代码自动构建部署 本地生成后手动上传
部署工具 GitHub Actions scp + Nginx
托管环境 GitHub 免费静态托管 自建云服务器 / 虚拟机
访问地址 用户名.github.io 服务器 IP / 域名
自主可控 依赖 GitHub 平台 完全自主可控

2. 核心命令与配置

命令 / 配置 作用
.github/workflows/deploy.yml 定义 CI/CD 流水线
secrets.PERSONAL_TOKEN 工作流跨仓库部署所需的访问令牌
hexo generate 将源文件编译为 public/ 静态文件
scp -r public/* 用户@IP:目录 将静态文件递归上传到服务器
Nginx root /var/www/blog 指定站点根目录

3. CI/CD 原理：CI（持续集成）指代码变更后自动构建，CD（持续部署）指构建通过后自动发布。GitHub Actions 在云端完成「检出 → 安装依赖 → 生成 → 部署」全过程，实现推送即上线，无需本地手动操作。

六、出现问题

问题现象 原因分析 解决方法
工作流部署失败 Token 未勾选 repo 权限 重新生成 Token 并勾选 repo
scp 提示主机指纹未知 首次连接服务器 输入 yes 确认后加入 known_hosts
访问服务器显示默认页 Nginx 站点配置未生效 检查 sites-enabled 链接并 reload

七、心得体会

通过本次实验，我完整走通了静态站点从开发到部署的两条路径。在技术层面，GitHub Actions 让我第一次直观体会到 CI/CD 的价值——只要 git push，构建和上线就自动完成，把重复劳动交给了流水线；而手动用 scp 部署到 Nginx 服务器，则让我理解了「静态文件 + Web 服务器」这一最朴素的托管本质，两相对比更能看清自动化与自主可控之间的取舍。

在思政层面，GitHub Actions、Nginx 等开源工具与免费平台让个人开发者也能低成本地实践工业级的部署流程，这正是开源生态的普惠价值。同时我也意识到，将服务长期托管在国外平台存在访问速度与自主可控的隐患，自建服务器和国产云平台是必要的补充。作为未来的软件工程师，我应在善用全球开源资源的同时，重视自主可控技术的建设。

图7：在 ~/.bashrc 末尾添加 PATH、自定义变量与别名

用 sudo nano /etc/profile.d/myenv.sh 新建脚本并写入 export MY_GLOBAL_VAR="全局变量"，通过 /etc/profile.d/ 目录对所有用户生效，是推荐的系统级配置方式。

五、知识总结

1. 常用命令

命令 作用
uname -a 查看内核与架构信息
lsb_release -a 查看发行版版本
id / groups 查看用户 ID 与所属组
env / echo $X 查看环境变量
export / unset 临时设置 / 删除变量
source 重新加载配置文件使其生效

2. 环境变量配置文件对比

配置文件 作用范围 加载时机 生效方式
~/.bashrc 当前用户 每次打开新终端 source ~/.bashrc
/etc/profile 所有用户 用户登录时 登录 / source
/etc/profile.d/*.sh 所有用户 用户登录时（推荐） 登录 / source
/etc/environment 所有用户 系统级，需重启 重启

3. 变量作用范围：export 直接设置的变量仅在当前会话有效；写入 ~/.bashrc 对当前用户永久生效；写入 /etc/profile.d/ 则对所有用户生效。

六、出现问题

问题现象 原因 解决方法
修改 ~/.bashrc 后不生效 未重新加载配置 执行 source ~/.bashrc
编辑 /etc/ 下文件被拒绝 普通用户权限不足 命令前加 sudo 提权

七、心得体会

通过本次实验，我熟悉了 Linux 命令行的基础操作，也理解了环境变量“临时”与“永久”、“用户级”与“系统级”之间的区别：export 只作用于当前会话，~/.bashrc 面向单个用户，而 /etc/profile.d/ 面向所有用户，配置的层级决定了它的作用范围。这种分层设计让系统在灵活性与统一管理之间取得了平衡。

在思政层面，Linux 作为开源操作系统的代表，其开放的源码和清晰的配置体系让每个使用者都能理解并掌控自己的系统，这正是开源精神的体现。作为未来的软件工程师，扎实掌握 Linux 环境配置是服务端开发与运维的基本功，也让我更加体会到自主可控的底层技术能力的重要性。

实验四 Web 站点构建 —— 实验报告

一、实验目的

本次实验旨在将实验二中已在本地搭建好的 Hexo 静态博客部署到 GitHub Pages，使其正式上线并可被公网访问，从而掌握静态站点“本地生成 → 远程部署 → 免费上线”的完整流程，理解静态站点生成器（SSG）的部署原理及其与动态站点的区别。

说明：实验四的步骤一至步骤六（环境准备、安装 Hexo、初始化项目、本地预览、编写文章、更换主题）已在实验二中完成，故本报告从步骤七：部署到 GitHub Pages 开始记录。

五、知识总结

1. 部署相关命令与配置

命令 / 配置 作用
npm install hexo-deployer-git 安装 Hexo 的 Git 部署插件
hexo clean 清理缓存与已生成的静态文件
hexo generate / hexo g 将源文件编译为静态 HTML
hexo deploy / hexo d 将 public/ 中的静态文件推送到远程仓库
_config.yml 的 deploy 项 指定部署方式、目标仓库地址与分支

2. GitHub Pages 部署要点

要点 说明
仓库命名 必须为 用户名.github.io，否则无法识别为个人站点
仓库可见性 需设为 Public
部署分支 与 _config.yml 中 branch 一致（本实验为 main）
访问地址 https://用户名.github.io

3. 静态站点部署原理：Hexo 在本地将 Markdown 与主题模板“提前编译”成纯静态文件（public/），部署插件再将这些文件推送到 GitHub 仓库；GitHub Pages 作为静态托管服务直接对外提供这些文件。整个过程无需服务器后端和数据库，这正是静态站点访问快、部署简单、安全性高的原因。

六、出现问题

问题现象 原因分析 解决方法
hexo deploy 提示找不到插件 未安装 hexo-deployer-git 执行 npm install hexo-deployer-git --save
部署时提示认证失败 未配置 Git 凭证或密码不再受支持 使用 Personal Access Token 代替密码，或配置 Git 凭证缓存
部署后页面 404 仓库名不是 用户名.github.io 检查并修正仓库命名
推送时出现 LF/CRLF 换行警告 Windows 与 Git 换行符差异 属正常提示，不影响部署，可用 git config core.autocrlf 调整

七、心得体会

通过本次实验，我完成了个人博客从本地到公网的最后一步——部署上线。在技术层面，我理解了静态站点部署的本质：本地 hexo generate 生成的其实就是一堆纯静态文件，部署插件所做的只是把它们推送到一个能对外提供文件访问的仓库，GitHub Pages 则免费承担了托管的角色。相比动态站点需要配置服务器和数据库，这种“生成好再上传”的方式让个人开发者也能零成本拥有一个稳定、快速的网站。

在思政层面，Hexo、Git 以及 GitHub Pages 提供的免费托管服务，让每个学生都能把自己的作品发布到互联网上，这体现了开源生态与开放平台带来的普惠价值。同时我也认识到，个人站点长期依赖国外平台存在访问速度和自主可控方面的隐患，国内的 Gitee Pages 等平台正是有益的补充。作为未来的软件工程师，我应当在善用全球开源资源的同时，关注并支持国产平台与自主可控技术的发展。

实验二报告

实验二 Web软件开发环境 —— 实验报告

一、实验目的

搭建 Web 开发环境（Node.js、npm、Git），安装并运行 Hexo 静态博客框架，完成博客初始化、写文章、本地预览与静态页面生成的完整流程，理解静态站点与动态站点的区别。

二、实验环境

  • 操作系统：Windows 11 家庭中文版（10.0.26100）
  • Node.js v24.14.1 / npm 11.12.1 / Git 2.53.0.windows.2
  • hexo-cli 4.3.2 / Hexo 7.3.0，主题 landscape（默认）
  • 编辑器：VS Code；终端：PowerShell

三、实验过程

步骤一：安装并验证 Node.js 与 npm　安装 LTS 版本后执行 node --version、npm --version 验证。


图1：Node.js v24.14.1、npm 11.12.1

步骤二：安装 Git　执行 git --version 验证，并用 git config --global 配置用户名与邮箱。


图2：Git 2.53.0.windows.2

步骤三：安装 Hexo CLI　执行 npm install -g hexo-cli，再用 hexo -v 查看版本及依赖。


图3：hexo-cli 4.3.2

步骤四：初始化并预览博客　执行 hexo init my-blog 初始化项目，在项目目录下 hexo server 启动本地服务，浏览器访问 http://localhost:4000 查看默认首页。


图4：hexo server 运行于 localhost:4000


图5：默认 Hexo 博客首页（landscape 主题）

验证项 操作 结果

最终成果：一个可本地访问的 Hexo 静态博客，包含自行发布的原创文章，源码已纳入 Git 管理。

五、知识总结

Web 开发环境核心组件：Node.js（运行时）、npm（包管理）、Git（版本控制）、Hexo（构建工具）。

Hexo 常用命令：hexo init（初始化）、hexo new（新建文章）、hexo generate/g（生成静态文件）、hexo server/s（本地预览）、hexo clean（清理缓存）。

静态 vs 动态博客：静态站点在构建时预先生成 HTML，无需数据库、访问快、安全性高（Hexo）；动态站点在访问时由服务器实时生成，功能丰富但需数据库（WordPress）。

Front-matter：Markdown 开头 --- 之间的区域，用于定义文章元数据（title、date、tags 等）。

问题 解决方法

hexo 命令无法识别 全局安装 npm install -g hexo-cli 并重启终端
改文章后页面未更新 重新执行 hexo generate 生成静态文件
