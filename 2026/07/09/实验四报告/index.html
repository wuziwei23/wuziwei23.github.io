实验四报告

实验四 Web 站点构建 —— 实验报告

一、实验目的

本次实验旨在将实验二中已在本地搭建好的 Hexo 静态博客部署到 GitHub Pages，使其正式上线并可被公网访问，从而掌握静态站点“本地生成 → 远程部署 → 免费上线”的完整流程，理解静态站点生成器（SSG）的部署原理及其与动态站点的区别。

说明：实验四的步骤一至步骤六（环境准备、安装 Hexo、初始化项目、本地预览、编写文章、更换主题）已在实验二中完成，故本报告从步骤七：部署到 GitHub Pages 开始记录。

二、实验环境

  • 操作系统：Windows 11 家庭中文版（内部版本号 10.0.26100）
  • 构建工具：hexo-cli 4.3.2，Hexo 7.3.0
  • 博客主题：landscape（Hexo 默认主题）
  • 部署插件：hexo-deployer-git
  • 版本控制：Git 2.53.0.windows.2
  • 代码托管 / 部署平台：GitHub Pages（仓库 wuziwei23.github.io）
  • 终端：Windows PowerShell
  • 网络：可访问互联网

五、知识总结

1. 部署相关命令与配置

命令 / 配置 作用
npm install hexo-deployer-git 安装 Hexo 的 Git 部署插件
hexo clean 清理缓存与已生成的静态文件
hexo generate / hexo g 将源文件编译为静态 HTML
hexo deploy / hexo d 将 public/ 中的静态文件推送到远程仓库
_config.yml 的 deploy 项 指定部署方式、目标仓库地址与分支

2. GitHub Pages 部署要点

要点 说明
仓库命名 必须为 用户名.github.io，否则无法识别为个人站点
仓库可见性 需设为 Public
部署分支 与 _config.yml 中 branch 一致（本实验为 main）
访问地址 https://用户名.github.io

3. 静态站点部署原理：Hexo 在本地将 Markdown 与主题模板“提前编译”成纯静态文件（public/），部署插件再将这些文件推送到 GitHub 仓库；GitHub Pages 作为静态托管服务直接对外提供这些文件。整个过程无需服务器后端和数据库，这正是静态站点访问快、部署简单、安全性高的原因。

六、出现问题

问题现象 原因分析 解决方法
hexo deploy 提示找不到插件 未安装 hexo-deployer-git 执行 npm install hexo-deployer-git --save
部署时提示认证失败 未配置 Git 凭证或密码不再受支持 使用 Personal Access Token 代替密码，或配置 Git 凭证缓存
部署后页面 404 仓库名不是 用户名.github.io 检查并修正仓库命名
推送时出现 LF/CRLF 换行警告 Windows 与 Git 换行符差异 属正常提示，不影响部署，可用 git config core.autocrlf 调整

七、心得体会

通过本次实验，我完成了个人博客从本地到公网的最后一步——部署上线。在技术层面，我理解了静态站点部署的本质：本地 hexo generate 生成的其实就是一堆纯静态文件，部署插件所做的只是把它们推送到一个能对外提供文件访问的仓库，GitHub Pages 则免费承担了托管的角色。相比动态站点需要配置服务器和数据库，这种“生成好再上传”的方式让个人开发者也能零成本拥有一个稳定、快速的网站。

在思政层面，Hexo、Git 以及 GitHub Pages 提供的免费托管服务，让每个学生都能把自己的作品发布到互联网上，这体现了开源生态与开放平台带来的普惠价值。同时我也认识到，个人站点长期依赖国外平台存在访问速度和自主可控方面的隐患，国内的 Gitee Pages 等平台正是有益的补充。作为未来的软件工程师，我应当在善用全球开源资源的同时，关注并支持国产平台与自主可控技术的发展。
